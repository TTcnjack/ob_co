## 核心概念

Superpowers 是一个完整的软件开发工作流系统，专为 AI 编码助手设计。它基于一组可组合的"技能"（Skills）构建，通过自动触发机制确保 AI 助手在正确的时机使用正确的技能。

### 设计理念

- **技能驱动**: 将开发流程分解为可复用的技能模块
- **自动触发**: AI 助手自动检测并激活相关技能，无需手动调用
- **系统化流程**: 用结构化流程替代临时性决策
- **测试优先**: 强制执行 TDD（测试驱动开发）
- **子代理协作**: 利用多个 AI 代理并行工作，提高效率

### 工作原理

Superpowers 改变了传统的 AI 编码流程：

```
传统流程: 需求 → 直接写代码 → 调试 → 完成

Superpowers 流程:
需求 → 头脑风暴（设计） → 审查设计 → 制定计划 → 
子代理执行 → 代码审查 → 测试验证 → 完成
```

**关键特点**:
- AI 不会立即开始写代码，而是先理解需求
- 通过对话提炼出完整的规格说明
- 分段展示设计文档，便于审查
- 生成详细的实施计划
- 使用子代理并行执行任务
- 每个步骤都有验证机制

## 主要功能

### 1. 自动技能触发

**工作机制**:
- AI 助手在每个任务前自动检查相关技能
- 根据上下文自动激活对应的技能
- 技能是强制性工作流，而非建议

**触发条件示例**:
- 开始编码前 → 激活 `brainstorming`
- 设计批准后 → 激活 `using-git-worktrees`
- 有实施计划时 → 激活 `subagent-driven-development`
- 实现功能时 → 激活 `test-driven-development`

### 2. 核心工作流

**完整开发周期**:

1. **brainstorming（头脑风暴）**
   - 在编写代码前激活
   - 通过提问细化粗略想法
   - 探索替代方案
   - 分段展示设计供验证
   - 保存设计文档

2. **using-git-worktrees（使用 Git 工作树）**
   - 设计批准后激活
   - 在新分支上创建隔离工作空间
   - 运行项目设置
   - 验证测试基线干净

3. **writing-plans（编写计划）**
   - 有批准的设计时激活
   - 将工作分解为小任务（每个 2-5 分钟）
   - 每个任务包含：确切文件路径、完整代码、验证步骤

4. **subagent-driven-development（子代理驱动开发）**
   - 有计划时激活
   - 为每个任务派发新的子代理
   - 两阶段审查：规格合规性 + 代码质量
   - 或使用 `executing-plans` 批量执行

5. **test-driven-development（测试驱动开发）**
   - 实现期间激活
   - 强制执行 RED-GREEN-REFACTOR 循环
   - 先写失败测试 → 看它失败 → 写最小代码 → 看它通过 → 提交
   - 删除在测试前编写的代码

6. **requesting-code-review（请求代码审查）**
   - 任务间激活
   - 根据计划审查代码
   - 按严重性报告问题
   - 关键问题阻止进度

7. **finishing-a-development-branch（完成开发分支）**
   - 任务完成时激活
   - 验证测试
   - 展示选项（合并/PR/保留/丢弃）
   - 清理工作树

### 3. 技能库系统

**技能组织**:
- 技能存储在独立的 `SKILL.md` 文件中
- 每个技能包含触发条件、执行步骤、验证标准
- 技能可以引用其他技能
- 支持自定义技能扩展

**技能更新**:
```bash
/plugin update superpowers
```

## 技能列表和说明

### 测试类技能

**test-driven-development（测试驱动开发）**
- **功能**: 强制执行 RED-GREEN-REFACTOR 循环
- **包含**: 测试反模式参考
- **触发**: 实现功能时自动激活
- **要求**: 先写测试，后写代码

### 调试类技能

**systematic-debugging（系统化调试）**
- **功能**: 4 阶段根因分析流程
- **包含**: 根因追踪、深度防御、基于条件的等待技术
- **触发**: 遇到 bug 时激活
- **方法**: 系统化而非猜测

**verification-before-completion（完成前验证）**
- **功能**: 确保问题真正修复
- **触发**: 声称修复完成前
- **要求**: 提供证据而非声明

### 协作类技能

**brainstorming（头脑风暴）**
- **功能**: 苏格拉底式设计细化
- **触发**: 开始编码前
- **输出**: 分段设计文档

**writing-plans（编写计划）**
- **功能**: 详细的实施计划
- **触发**: 设计批准后
- **输出**: 可执行的任务列表

**executing-plans（执行计划）**
- **功能**: 批量执行任务，带检查点
- **触发**: 有计划时
- **特点**: 人工检查点

**dispatching-parallel-agents（派发并行代理）**
- **功能**: 并发子代理工作流
- **触发**: 需要并行处理时
- **优势**: 提高执行效率

**requesting-code-review（请求代码审查）**
- **功能**: 预审查检查清单
- **触发**: 任务完成时
- **输出**: 问题报告

**receiving-code-review（接收代码审查）**
- **功能**: 响应反馈
- **触发**: 收到审查意见时
- **要求**: 系统化处理反馈

**using-git-worktrees（使用 Git 工作树）**
- **功能**: 并行开发分支
- **触发**: 设计批准后
- **优势**: 隔离工作空间

**finishing-a-development-branch（完成开发分支）**
- **功能**: 合并/PR 决策工作流
- **触发**: 任务完成时
- **选项**: 合并、PR、保留、丢弃

**subagent-driven-development（子代理驱动开发）**
- **功能**: 快速迭代，两阶段审查
- **触发**: 有计划时
- **审查**: 规格合规性 + 代码质量
- **特点**: 可自主工作数小时

### 元技能

**writing-skills（编写技能）**
- **功能**: 创建新技能，遵循最佳实践
- **包含**: 测试方法论
- **用途**: 扩展技能库

**using-superpowers（使用 Superpowers）**
- **功能**: 技能系统介绍
- **用途**: 新用户入门

## 子代理驱动开发

### 核心概念

**子代理驱动开发**是 Superpowers 的核心创新之一，允许主 AI 助手派发多个子代理并行工作。

### 工作机制

**流程**:
```
主代理 → 制定计划 → 派发子代理 → 并行执行 → 审查结果 → 继续
```

**特点**:
- 每个任务由独立的子代理执行
- 子代理专注于单一任务
- 两阶段审查机制：
  1. **规格合规性审查**: 是否符合计划要求
  2. **代码质量审查**: 代码是否优质

### 优势

**效率提升**:
- 并行处理多个任务
- 减少上下文切换
- 主代理保持高层视角

**质量保证**:
- 每个子代理都有明确的任务边界
- 强制审查流程
- 防止偏离计划

**自主性**:
- Claude 可以自主工作数小时
- 无需频繁人工干预
- 自动检测和修复问题

### 使用场景

**适合子代理驱动开发**:
- 大型功能开发
- 多个独立任务
- 需要长时间自主工作
- 任务可以并行化

**适合批量执行（executing-plans）**:
- 小型任务
- 需要频繁人工检查
- 任务间有强依赖关系

## 使用示例

### 基本工作流

**场景：开发新功能**

```
用户: "我想添加一个用户认证功能"

AI（自动激活 brainstorming）:
"让我先理解一下需求。这个认证功能需要支持哪些登录方式？
是否需要密码重置？会话管理的要求是什么？..."

[经过对话后]

AI: "我整理了设计文档，分为以下几个部分：
1. 认证流程设计
2. 数据模型
3. API 端点
4. 安全考虑
让我们逐一审查..."

[设计批准后，自动激活 using-git-worktrees]

AI: "我将在新分支 'feature/user-auth' 上创建工作树..."

[自动激活 writing-plans]

AI: "我制定了实施计划，包含 12 个任务：
Task 1: 创建 User 模型（2 分钟）
Task 2: 实现密码哈希（3 分钟）
..."

[自动激活 subagent-driven-development]

AI: "我将派发子代理执行这些任务。预计 30 分钟完成。"

[2 小时后]

AI: "所有任务完成。测试通过。准备合并到主分支。"
```

### 调试场景

**场景：修复 Bug**

```
用户: "登录功能有时会失败"

AI（自动激活 systematic-debugging）:
"让我系统化地调试这个问题。

阶段 1: 重现问题
- 我需要了解失败的具体场景...

阶段 2: 收集证据
- 检查日志...
- 分析错误模式...

阶段 3: 根因分析
- 发现问题：会话 token 在并发请求时冲突

阶段 4: 验证修复
- 实施修复...
- 运行测试..."

[自动激活 verification-before-completion]

AI: "修复已验证：
- 单元测试通过
- 集成测试通过
- 手动测试 100 次并发请求，无失败"
```

### 创建自定义技能

**场景：扩展技能库**

```
用户: "我想创建一个代码性能分析的技能"

AI（自动激活 writing-skills）:
"让我帮你创建这个技能。按照最佳实践：

1. 技能元数据
   - 名称: performance-profiling
   - 触发条件: 性能问题或优化请求

2. 技能结构
   - 触发器定义
   - 执行步骤
   - 验证标准
   - 测试方法

3. 创建 SKILL.md 文件..."
```

## 最佳实践

### 1. 信任自动触发

- **不要手动调用技能**: 让 AI 自动检测和激活
- **遵循工作流**: 不要跳过步骤（如跳过设计直接编码）
- **耐心等待**: 头脑风暴和计划阶段看似慢，但能节省后期时间

### 2. 充分利用设计阶段

- **详细描述需求**: 在头脑风暴阶段提供充分信息
- **仔细审查设计**: 分段审查设计文档，确保理解一致
- **提出问题**: 设计阶段是最便宜的修改时机

### 3. 编写清晰的计划

- **任务粒度**: 每个任务 2-5 分钟
- **明确验证**: 每个任务都有清晰的完成标准
- **包含测试**: 计划中明确测试要求

### 4. 拥抱测试驱动开发

- **先写测试**: 永远先写测试，后写代码
- **看测试失败**: 确保测试真的能捕获问题
- **最小实现**: 只写通过测试所需的最少代码
- **重构**: 测试通过后再优化代码

### 5. 有效使用子代理

- **大任务分解**: 将大功能分解为独立的小任务
- **明确边界**: 每个子代理任务有清晰的输入输出
- **信任审查**: 依赖两阶段审查机制保证质量
- **监控进度**: 定期检查子代理的执行状态

### 6. 代码审查

- **自动审查**: 利用 `requesting-code-review` 技能
- **关注关键问题**: 优先处理阻塞性问题
- **系统化反馈**: 使用 `receiving-code-review` 处理反馈

### 7. Git 工作流

- **使用工作树**: 利用 `using-git-worktrees` 隔离工作
- **频繁提交**: 每个通过的测试都提交
- **清晰的提交信息**: 描述做了什么和为什么

### 8. 扩展技能库

- **识别模式**: 发现重复的工作流程
- **创建技能**: 使用 `writing-skills` 技能创建新技能
- **测试技能**: 确保新技能有测试方法
- **分享技能**: 贡献回社区

## 核心哲学

### 测试驱动开发（TDD）

- 先写测试，永远
- RED-GREEN-REFACTOR 循环
- 测试是规格说明

### 系统化优于临时性

- 用流程替代猜测
- 可重复的方法
- 文档化的决策

### 复杂度降低

- 简单性是首要目标
- YAGNI（你不会需要它）
- DRY（不要重复自己）

### 证据优于声明

- 验证后再声明成功
- 测试证明功能
- 日志证明行为

## 故障排查

### 技能未自动触发

**可能原因**:
- 插件未正确安装
- 上下文不明确

**解决方案**:
- 验证安装：尝试手动触发技能
- 明确表达意图："让我们先设计这个功能"

### 子代理执行缓慢

**可能原因**:
- 任务粒度太大
- 任务间依赖过多

**解决方案**:
- 将任务分解得更小（2-5 分钟）
- 减少任务间的依赖关系

### 测试失败

**可能原因**:
- 跳过了 TDD 流程
- 测试编写不当

**解决方案**:
- 严格遵循 RED-GREEN-REFACTOR
- 确保测试先失败再通过

### 代码偏离计划

**可能原因**:
- 计划不够详细
- 跳过了审查步骤

**解决方案**:
- 编写更详细的计划
- 启用两阶段审查
